Split EDI

Splitting EDI into single-transaction interchanges

Split multi-transaction EDI interchanges into a series equivalent single transaction interchanges

An EDI interchange containing multiple transactions is split into a series of interchanges containing only a single transaction. Each interchange is written to a file having a name formed according to a namingPattern and placed in an outputDirectory. The interchange and functional group enveloping is preserved from the EDI input, except that the functional group transaction count is adjusted appropriately.

query Parameters
input
string

Path to EDI to be split. If omitted, the input EDI is taken from the request body.

namingPattern
string

Pattern for naming the split files. The pattern is simply a filename including a sequence of 4 or more 0s indicating where a sequence number is placed to achieve uniqueness. The default is split-0000.edi. Using this an example, the produced split filess will be named split-0001.edi, split-0002.edi, etc.

outputDirectory
string

Path for the directory where split files are to be written. Remember that each / in the path must be percent-encoded; each / is expressed as %2F. For example, instead of outputDirectory=/tmp you should use outputDirectory=%2Ftmp.

Request Body schema:
required
string

Create EDI Samples

Creating EDI samples from an EDI model

Create an EDI sample

Creates an EDI interchange containing a sample transaction of the specified type and version. This operation requires a corresponding model definition.

The generated sample includes every permitted segment type for each defined loop in the transaction. Each segment contains representative values for all elements, respecting the elementâ€™s data type and length constraints. For elements with defined code lists, values are randomly selected from those lists.

Where applicableâ€”such as for street names, cities, states, or ZIP codesâ€”realistic but non-identifying values are used. While the result does not resemble an actual business transaction, it provides a valid, standards-compliant EDI example suitable for testing or demonstration purposes without exposing private information.

List EDI Models

Reporting which EDI models are currently installed

Returns a list of the EDI models currently installed.

An EDI model defines the structure, compliance rules, and other metadata or a specific EDI document type and version. Examples include 850 version 004010 for an X12 Purchase Order and ORDERS release D96A for an EDIFACT Purchase Order.
